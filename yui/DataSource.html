Yahoo! UI Library: DataSource

DataSource

The DataSource Utility provides a common configurable interface for other components to interact with different types of data, from simple JavaScript arrays to online servers over XHR.

Notes:

  1. The DataSource Utility is in beta release; please refer to the YUI FAQ entry on beta releases for more information about what the beta designation implies.
  2. YUI contains two DataSource components. YAHOO.util.DataSource, documented here, is intended to be the foundational class for the library. YAHOO.widget.DataSource, which is distributed as part of the AutoComplete Control, will be deprecated in favor of YAHOO.util.DataSource in a future release.

Usage

The DataSource is capable of fetching tabular data from a variety of local or remote sources. It is most often used with the DataTable component in order to retrieve the data for it to show and future releases of the AutoComplete control will also use it.

The DataSource will call the callback function provided with an object containing the status of the request and, if successful, an array containing the data retrieved. It has the capability of going deep into the hierarchy of the source data, select specified fields from the raw input data, parse them as indicated and call the provided callback function when finished.

The DataSource has an optional cache to store retrieved and parsed data. It can also be set to periodically poll for data.

Instantiating a DataSource

The first mandatory argument to instantiate a DataSource is where it should fetch the data from. It can be any one of several types.

String
It will assume it to be a base URL for a remote server that will provide the data.
XML object
A reference to an XML DOM object
HTML table reference
A reference to a <table> HTML element
Array
A native JavaScript array.
Function reference
Reference to a function that will provide the data.
Object
An object that, at some depth, has an array of data

The second optional argument, if present, should be an object that allows setting the properties of the DataSource in a single step at creation time. All properties are accessible though they can later be modified by explicit assignments, except dataType, which will be determined by the nature of the first argument.

When providing a URL it is important that it has a trailing ? or & for the arguments of the URL (first argument of sendRequest, see below) to be appended successfully, at least for a GET request. If set to do POST requests such trailing character won't hurt so it is better to have it there just in case you switch methods.

Using the DataSource

Once created and configured, the DataSource can be used to retrieve data directly or it can be passed to another component (currently the DataTable component) that will use it to fetch the data it needs.

Properties

connMethodPost

This boolean property is only meaningful when the data comes from a remote source. By default it is set to false and the HTTP request will be sent as a GET request. If set to true, the request will be sent as a POST.

responseType

It tells the DataSource the type of data that is to be read. It is particularly important for remote connections or when the data is provided by a function since those could return data in any of the supported formats. Nevertheless it is required in all other formats as well. It should be set to any of the following constants. They all should be specified in full with the leading YAHOO.util.DataSource component name.

  • TYPE_HTMLTABLE
  • TYPE_JSARRAY
  • TYPE_JSON
  • TYPE_TEXT
  • TYPE_XML

The same constants are used internally for the dataType property and will be set by the type of the first argument of the constructor. There are further constants defined that only are useful for the dataType property.

responseSchema

The responseSchema property is an object containing an assortment of further properties describing the incoming data which depends on the type of that data.

resultsList
A dot separated sequence of identifiers used to reach the data (valid for TYPE_JSON).
resultNode
The name of the XML elements containing tabular data to be read (valid for TYPE_XML).
recordDelim, fieldDelim
Characters or strings used to delimit records and fields within the record in data of TYPE_TEXT.
fields
Array of field names or object literals of the format {key: , parser: } indicating the name of each field and the parser to be used.

The fields property is required for all responseTypes. It is an array that can have any mix of strings with the names of the fields to be retrieved or object literals with a key property giving the field name and the parser property that can be a reference to any of the supplied parse functions or to a custom-made parse function. The parser will receive a single argument which is the raw field value to be parsed and should return it parsed.

The DataSource will either pick the values of the elements named in the fields array, no matter in which order they are found within the data or will assign that name to unnamed elements in the order they are found. Fetching Data

You can retrieve data by calling the sendRequest method. It requires three arguments:

oRequest
If the first argument to the constructor was a string, it is assumed to be a URL and this argument can either be a URL-encoded list of arguments to be appended to the base URL (for a GET request) or anything valid to be sent in the document body section of the request for a POST request when connMethodPost = true.
If the first argument was a function reference, then this can be anything that can be passed as a single argument to that function.
Note: this is where the initialRequest configuration parameter of the DataTable component will go.
oCallback
A reference to the function that will be called back when the data is ready. This function will receive the following arguments:
oRequest
The same value as passed in the first argument to sendRequest.
oParsedResponse
An object containing either a single property error set to true if any error occurred while retrieving or parsing the data or a results property with an array with the requested data
scope
The scope to be used when calling the callback function.

Polling

The setInterval method can be used to repeatedly check the source of data. The first argument is the number of milliseconds to wait between successive polls, the following three arguments are the same as those of sendRequest. This method returns a numeric identifier that can be used to cancel the polling.

The polling can be cancelled by calling clearInterval and passing it the identifier returned by the call to setInterval. All polling can be cancelled by calling clearAllIntervals .

Caching

The DataSource component can cache data. You can set the maxCacheEntries property to the number of responses you want cached. When this number is exceeded, the oldest response cached will be dropped. The data in the cache is already fully processed, extracted, parsed and arranged ready to be returned to the caller. On a cache hit, those events or overridable methods meant to signal or process the retrieval of data will not be fired or called.

Currently, the maxCacheEntries property has to be set in the configuration options (second argument) of the constructor of the DataSource.

The cache will be updated when polling but won't be used for retrieval.

Cache entries are indexed by the oRequest (first) argument of sendRequest, there will be one entry per request up to the total of maxCachedEntries.

Interesting Times

Several events can be used to monitor the progress of the retrieval of data or its result. There are also overridable methods or plug-in parsers that are meant to transform the data before it is returned to the requesting application.

requestEvent
Its most useful argument is oArgs.response which is the raw response (undecoded) as received from the original source. This is useful to retrieve any extra information (meta-data) contained along the main tabular data.
doBeforeParseData
If the data is formatted in a way that the DataSource would be unable to parse, this method can be overridden to fix it. This method is meant to fix the structure of the data itself or resolve a very complex hierarchy. For example, the DataSource cannot parse fields in an XML document that are not at the same level within the node named with the resultNode property or with the resultsList property for JSON data. On the other hand, it will retrieve all the nodes of the given name at any level even when nested one into the other and you might want to do a shallow traversal of the XML document.

This method receives two arguments, the same oRequest value as passed in the first argument to sendRequest and the raw data as just retrieved without any parsing or any other processing. It should return the processed data.

If the data is to be heavily re-structured by this method, it is usually a good idea to declare the responseType as TYPE_JSARRAY since that is the type that requires less processing later on and it is far easier than re-creating, for example, an XML document or assembling a JSON-encoded string.

doBeforeCallback
This method will be called after the data was successfully parsed but before the data is added to the cache or the callback called. Its first two arguments are the same as for the previous function. The third is the already parsed response. Whatever this method returns is what the callback function will get. If nothing else, the original parsed response, the third argument, should be returned.

These two methods offer a better performance than listening to requestEvent when reading extra information from the received response though care should be taken not to tamper with the data.

Parsers
Parser functions can be designated for individual fields in the fields array of the responseSchema. The DataSource will call the parser for each record or the data retrieved and will pass it the raw value of that field and expects the parsed value in return. You can use the built-in parsers, parseDate which will handle any date that the native JavaScript Date.parse() method handles (notice not all date formats are supported) or parseNumber which will try to parse a number whether an integer or float.

Feeding the DataTable

The DataTable provides three functions that can be used as callbacks on a call to sendRequest:

onDataReturnInitializeTable
Draws a DataTable with the data provided. On an existing DataTable it will delete the current contents before drawing the new. This is the method used by DataTable itself on initializing a newly created DataTable. It is the method that would be used with polling to refresh the DataTable with new data.
onDataReturnInsertRows
This method will insert new rows at the beginning of the DataTable, preserving the existing data and regardless of any sort order.
onDataReturnAppendRows
Rows will be added to the end of the DataTable, preserving the existing records, regardless of sort order.
